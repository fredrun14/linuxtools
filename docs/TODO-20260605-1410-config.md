## Contexte

### Problématique
`config/manager.py` (méthode `_write_toml_section`, l.245-264) génère un TOML
**invalide** :
- une `list` Python est sérialisée via `str()` → `['a', 'b']` (TOML exige
  `["a", "b"]`, guillemets doubles) ;
- les chaînes ne sont pas échappées → un `"` ou `\n` dans la valeur casse le
  fichier.

Or le projet possède déjà `ConfTomlExporter` (module `dotconf`, corrigé au
commit `cc5d062` pour l'échappement des caractères de contrôle). Maintenir deux
exporteurs TOML divergents est la cause racine.

### Solution technique retenue
Déléguer la sérialisation TOML à `ConfTomlExporter` plutôt que de corriger un
second sérialiseur maison. Élimine la duplication ET le bug.

### Fichiers impactés
- `src/linux_python_utils/config/manager.py` — délégation TOML + except ciblé
- `src/linux_python_utils/config/loader.py` — suppression code mort + typing
- `src/linux_python_utils/config/xdg.py` — création atomique du fichier
- `tests/test_config.py` — cas TOML adversariaux

### `config/xdg.py` — 🟡 MINEUR

#### Logique
`init_config_file` : remplacer le `check-then-write` (`exists()` puis
`write_text`) par une ouverture exclusive atomique :

```python
try:
    with open(path, "x", encoding="utf-8") as f:  # échoue si existe
        f.write(template)

    pass  # fichier déjà présent : ne rien écraser
```

Si le fichier peut contenir des secrets : ajouter `os.chmod(path, 0o600)`.
