## Contexte

### Problématique
1. `FileLogger` crée le fichier de log avec les permissions par défaut (umask,
   souvent `0o644`) et sans `O_NOFOLLOW`. Le log peut contenir des évènements
   `SecurityLogger` sensibles, lisibles par tous, et est vulnérable à une
   attaque par symlink pré-positionné.
2. `SecurityLogger` sérialise `details` sans masquer les clés sensibles
   (`password`, `token`, etc.) → vecteur de fuite.
3. `TeeStream.__getattr__` délègue à `_original` → `close()` ne ferme jamais
   `_log_fh` (descripteur fuité, log non flushé).

### Solution technique retenue
Créer le fichier log via `os.open(O_CREAT|O_WRONLY|O_APPEND|O_NOFOLLOW, 0o600)`
et brancher ce descripteur sur le `FileHandler`. Masquer les clés sensibles avant
`json.dumps`. Définir un `close()` explicite sur `TeeStream`.

### Fichiers impactés
- `src/linux_python_utils/logging/file_logger.py` — permissions + validation level
- `src/linux_python_utils/logging/security_logger.py` — redaction `details`
- `src/linux_python_utils/logging/tee_stream.py` — `close()` explicite
- `tests/test_logging.py` — test permissions + TeeStream

## Évolutions à mettre en place (Détail Junior)

### `file_logger.py` — 🔴 BLOQUANT

#### Étape 1 — Ouverture sécurisée du fichier log
Créer un helper qui ouvre le fichier avec `O_NOFOLLOW` et mode `0o600`, puis le
fournir au `FileHandler` :

```python
import os
import logging

def _open_secure(path: str) -> int:
    """Ouvre le log en O_NOFOLLOW, mode 0o600 (anti-symlink/anti-lecture)."""
    flags = os.O_CREAT | os.O_WRONLY | os.O_APPEND | os.O_NOFOLLOW
    return os.open(path, flags, 0o600)

# usage :
handler = logging.FileHandler(log_file, delay=True)
# brancher le fd sécurisé :
handler.stream = os.fdopen(_open_secure(log_file), "a", encoding="utf-8")
```

> Variante plus simple si `FileHandler` pose problème : ouvrir le fd, créer un
> `logging.StreamHandler(os.fdopen(fd, "a"))`. Choisir une seule approche et la
> documenter.

#### Étape 2 — Même traitement pour `log_to_file` (l.150-158)
Remplacer `open(self.log_file, "a")` par `os.fdopen(_open_secure(...), "a")`.

#### Étape 3 — Valider le niveau de log (l.98)
```python
_NIVEAUX = {"DEBUG", "INFO", "WARNING", "ERROR", "CRITICAL"}

niveau = log_level_str.upper()
if niveau not in _NIVEAUX:
    raise ValueError(f"Niveau de log invalide : {log_level_str!r}")
level = getattr(logging, niveau)
```

#### Gestion d'erreurs
| Cas | Condition | Action |
|---|---|---|
| Cible est un symlink | `O_NOFOLLOW` → `OSError` (ELOOP) | propager (refus volontaire) |
| Niveau invalide | hors `_NIVEAUX` | `raise ValueError` |
| Logger déjà configuré | même chemin | documenter le partage (l.124-125) |

### `security_logger.py` — 🟠 MAJEUR

#### Logique
Avant `json.dumps(details)`, masquer les clés sensibles :

```python
_CLES_SENSIBLES = {"password", "passwd", "token", "secret",
                   "key", "authorization", "api_key"}

def _masquer(details: dict[str, Any]) -> dict[str, Any]:
    """Remplace la valeur des clés sensibles par '***'."""
    return {
        k: ("***" if k.lower() in _CLES_SENSIBLES else v)
        for k, v in details.items()
    }
```

Optionnel : remplacer `severity: str` par `Literal["info","warning","error",
"critical"]` ou un `StrEnum`.

### `tee_stream.py` — 🟠 MAJEUR

#### Logique
Définir `close()` explicitement (ne pas le laisser au `__getattr__`) :

```python
def close(self) -> None:
    """Ferme le flux de log ; ne ferme PAS le flux original (stdout)."""
    try:
        self._log_fh.flush()
        self._log_fh.close()
    except (OSError, ValueError):
        pass  # best-effort
```

Optionnel : isoler l'écriture log dans `write()` (l.61-62) dans un try/except
best-effort pour ne pas casser stdout/stderr de l'application si le disque est
plein.

## Points d'attention
- `delay=True` sur `FileHandler` évite d'ouvrir le fichier deux fois.
- Mode `0o600` peut gêner si plusieurs utilisateurs partagent un log : choisir
  `0o640` + groupe dédié seulement si le besoin est documenté.
